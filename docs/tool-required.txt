───────────────────────────────────────────────────────────────────────────────
  REDIS — DO YOU NEED IT?  YES, GENUINELY
───────────────────────────────────────────────────────────────────────────────

  THE PROBLEM IT SOLVES:
  ----------------------
  Every single song that finishes, the stream worker must immediately know:
    - What is the next track?
    - What is its file path?

  If you used PostgreSQL for this:
    Song ends -> DB query -> network round trip -> 5-50ms delay
    That gap is silence in the stream. Listeners hear a cut.

  Redis answers in under 1ms because it lives in RAM.

  THE SECOND PROBLEM IT SOLVES:
  ------------------------------
  You run 2 app replicas (for uptime/scaling).
  Both replicas need to share the SAME playlist state.
  You cannot store playlist in a variable inside NestJS because:

    Replica 1 RAM: playlist = [song1, song2, song3], index = 5
    Replica 2 RAM: playlist = [song1, song2, song3], index = 5
                                                              ^
                                                    DIFFERENT — out of sync

  Redis is shared memory across all replicas.
  One source of truth. Both replicas read/write the same list.

  WITHOUT REDIS YOU WOULD NEED:
  ------------------------------
  Store playlist in PostgreSQL and query it per song.
    -- Slow (adds silence between tracks)
    -- Works fine for 1 replica but breaks for 2+

  VERDICT:
  ---------
  Redis is the right tool here. Not optional if you want
  gapless playback and multi-replica support.

  WHAT REDIS IS STORING:
  ----------------------

  radio:playlist:active    LIST    [uuid1, uuid2, uuid3 ...]
  radio:playlist:staged    LIST    [uuid4, uuid5 ...]  <- cron builds this
  radio:playlist:idx       STRING  "42"                <- current position
  radio:playlist:meta      HASH    uuid1 -> { filePath, title, artist }
  radio:playlist:swap_lock STRING  "1" (expires in 30s) <- prevents race

───────────────────────────────────────────────────────────────────────────────
  BULLMQ — DO YOU NEED IT?  YES, GENUINELY
───────────────────────────────────────────────────────────────────────────────

  THE PROBLEM IT SOLVES:
  ----------------------
  The stream is a forever-running loop.
  It never returns. It just pipes audio until you kill it.

  If you ran it directly inside the HTTP handler:

    POST /radio/start
          |
          v
    radioService.startStream()  <-- this never returns
          |
          | HTTP request times out after 30s
          | Client gets a timeout error
          | NestJS thread is blocked forever
          | No other requests can be served

  BullMQ moves the stream into a background worker process.
  The HTTP handler returns in milliseconds.
  The worker runs the stream forever in the background.

  THE SECOND PROBLEM IT SOLVES:
  ------------------------------
  What happens if Icecast crashes and restarts?
  Your stream connection drops.

  Without BullMQ: stream is dead, you manually restart
  With BullMQ:    job retries automatically with exponential backoff

    Attempt 1 fails -> wait 3s  -> retry
    Attempt 2 fails -> wait 6s  -> retry
    Attempt 3 fails -> wait 12s -> retry
    Attempt 4 fails -> wait 24s -> retry
    Attempt 5 fails -> mark as failed, alert you

  THE THIRD PROBLEM IT SOLVES:
  ------------------------------
  You have 2 replicas. Both could try to start a stream.
  That means 2 simultaneous streams to Icecast. Chaos.

    concurrency: 1  on the worker means only ONE job runs at a time
                    across ALL replicas sharing the same Redis queue.

  WITHOUT BULLMQ YOU WOULD NEED:
  --------------------------------
  Run stream in a setInterval or background async function.
  Manually implement retry logic.
  Manually implement single-stream locking across replicas.
  Lose the stream silently when the process crashes.

  VERDICT:
  ---------
  BullMQ is the right tool. It is also built on Redis,
  so you are not adding a new infrastructure dependency.

───────────────────────────────────────────────────────────────────────────────
  KAFKA — DO YOU NEED IT?  HONESTLY, NOT RIGHT NOW
───────────────────────────────────────────────────────────────────────────────

  WHAT IT DOES IN THIS SYSTEM:
  -----------------------------
  Every time something happens, we emit an event:

    Stream started  ->  radio.events topic
    Song changed    ->  radio.now_playing topic
    Playlist built  ->  radio.events topic

  Right now nobody is consuming these events.
  They go into Kafka and sit there.
  The radio works perfectly without them.

  WHY IT WAS ADDED:
  ------------------
  It is a foundation for things you will likely want later:

    NOW PLAYING display on a website
    -- Consumer reads radio.now_playing
    -- Pushes to WebSocket
    -- Website shows "Now Playing: Artist - Song"

    Analytics dashboard
    -- Consumer reads radio.now_playing
    -- Counts plays per track per hour
    -- Feeds a chart

    Discord / Telegram bot
    -- Consumer reads radio.now_playing
    -- Posts "Now playing" message to your server

    Recommendation engine
    -- Consumer reads play history
    -- Adjusts track weights in PostgreSQL
    -- Next shuffle is smarter

  WHY KAFKA SPECIFICALLY AND NOT SOMETHING SIMPLER:
  --------------------------------------------------

    Option A: Redis Pub/Sub
    ----
    Simple. Already have Redis.
    BUT: messages are lost if no consumer is connected.
         Cannot replay history.
         Cannot have multiple independent consumers.

    Option B: A simple webhook / HTTP call
    ----
    Very simple.
    BUT: if the target service is down, event is lost.
         Tight coupling. Radio waits for HTTP response.

    Option C: Kafka
    ----
    Events are persisted on disk (default 7 days).
    Any number of consumers can read independently.
    Consumers can replay from any point in history.
    Radio does not wait for consumers. Fire and forget.
    Consumer crashes and restarts — picks up where it left off.

  SHOULD YOU KEEP KAFKA RIGHT NOW:
  ----------------------------------

    You have 0 consumers         ->  Kafka adds complexity for zero gain
    You have 1 consumer          ->  Redis Pub/Sub is simpler and enough
    You have 2+ consumers        ->  Kafka starts making sense
    You want replay / history    ->  Kafka is the right choice
    You are solo / small team    ->  Remove Kafka, add it later when needed

  HOW TO REMOVE KAFKA IF YOU WANT:
  ----------------------------------
  Delete all  this.kafka.emit(...)  calls.
  Remove KafkaModule import from app.module.ts.
  Remove kafkajs and @nestjs/microservices from package.json.
  Everything else works identically.

  VERDICT:
  ---------
  Kafka is optional right now. It is infrastructure for your future
  features, not a current requirement. Keep it if you plan to build
  a now-playing display or analytics. Remove it if you want simplicity.

───────────────────────────────────────────────────────────────────────────────
  POSTGRESQL — DO YOU NEED IT?  YES, OBVIOUSLY
───────────────────────────────────────────────────────────────────────────────

  Stores your music library permanently.
  Survives restarts, crashes, deploys.
  Lets you query: top played, by genre, weighted shuffle.
  Tracks play counts so the system gets smarter over time.

  You cannot store this in Redis because Redis is not
  designed for permanent relational data.
  You cannot store this in Kafka because Kafka is a log,
  not a queryable database.

  VERDICT:  Non-negotiable.

───────────────────────────────────────────────────────────────────────────────
  ICECAST — DO YOU NEED IT?  YES
───────────────────────────────────────────────────────────────────────────────

  NestJS cannot efficiently serve audio to many listeners.
  If 500 people connect to /radio/stream directly on NestJS:

    NestJS opens 500 file read streams simultaneously
    500 concurrent connections holding HTTP responses open
    Memory explodes
    CPU spikes

  Icecast is purpose-built for this.
  NestJS sends ONE stream to Icecast.
  Icecast fans it out to 500 listeners efficiently.
  That is its entire job and it does it very well.

  VERDICT:  Non-negotiable for production.

═══════════════════════════════════════════════════════════════════════════════
  FINAL SUMMARY — NEED vs NICE TO HAVE
═══════════════════════════════════════════════════════════════════════════════

  Tool          Need it?     Why
  ----------    ---------    -------------------------------------------------
  PostgreSQL    YES          Permanent music library, play counts, queries
  Redis         YES          Sub-millisecond playlist reads, shared state
  BullMQ        YES          Background worker, auto-retry, single stream lock
  Icecast       YES          Multi-listener audio broadcast
  Kafka         OPTIONAL     Future analytics/now-playing — zero consumers now

  Minimum viable version = PostgreSQL + Redis + BullMQ + Icecast
  Full production version = add Kafka when you have something consuming it
